## 2. 核心概念

1. 最小 `coding agent` 的核心不是复杂工作流，而是一个持续把工具结果喂回模型的 `while True` 循环。
2. 模型负责决策：是否调用工具、调用哪个工具、拿到结果后是否继续。
3. `Harness` 负责执行：接收 `tool_use`、运行本地工具、把 `tool_result` 放回 `messages`。
4. `stop_reason == "tool_use"` 表示模型还需要行动，循环继续；否则说明模型已经给出最终回答，循环退出。
5. `run_bash(command)` 是这一章唯一的工具执行器，用来把模型生成的 `shell` 命令变成真实世界的动作。

## 3. 关键代码

1. `agent_loop(messages)` 的核心结构：

```python
while True:
    response = client.messages.create(
        model=MODEL,
        system=SYSTEM,
        messages=messages,
        tools=TOOLS,
        max_tokens=8000,
    )
    messages.append({"role": "assistant", "content": response.content})

    if response.stop_reason != "tool_use":
        return
```

2. 工具执行和结果回填：

```python
results = []
for block in response.content:
    if block.type == "tool_use":
        output = run_bash(block.input["command"])
        results.append({
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": output,
        })
messages.append({"role": "user", "content": results})
```

3. `run_bash(command)` 的教学级保护：

```python
dangerous = ["rm -rf /", "sudo", "shutdown", "reboot", "> /dev/"]
```

## 4. 我理解的流程

```text
user message
  -> client.messages.create(...)
  -> model 返回 response.content
  -> 如果 stop_reason 不是 tool_use，返回最终回答
  -> 如果有 tool_use，读取 block.input["command"]
  -> run_bash(command)
  -> 组装 tool_result
  -> messages.append({"role": "user", "content": results})
  -> 回到下一轮模型调用
```

## 5. 需要注意的风险

1. `run_bash(command)` 使用 `shell=True`，只是教学 `demo`，不是生产级权限系统。
2. 当前危险命令拦截依赖字符串包含判断，只能挡住非常明显的危险命令。
3. 这一章只有 `bash` 一个工具，读写文件都要通过 `shell` 完成，所以后续 `s02_tool_use` 会把工具拆成更清晰的 `read`、`write`、`edit` 等接口。

## 6. 仍然不清楚的问题

1. 真实生产级 `agent loop` 为什么不能只依赖 `stop_reason`，需要在后续章节或源码对照中继续理解。
2. 权限检查应该放在工具执行前、工具执行器内部，还是更外层的统一策略层，需要等 `s03_permission` 再系统整理。

## 2. 核心概念

1. `OpenAI` 版本和 `Anthropic` 版本的 `Harness` 形状相同：模型决定是否调用工具，代码执行工具，再把结果回填给模型。
2. `Anthropic` 版通过 `tool_use` 和 `tool_result` 串联工具调用。
3. `OpenAI Responses API` 版通过 `function_call` 和 `function_call_output` 串联工具调用。
4. `call_id` 必须和模型刚才发出的工具调用对应，模型才能知道某个输出属于哪次 `function_call`。
5. `strict: True` 和 `additionalProperties: False` 用来约束模型按 `JSON Schema` 生成工具参数。
6. `response.output` 中的 `SDK` 对象需要转换为普通 `dict` 后再进入下一轮 `input`，所以代码引入了 `as_input_item(item)`。

## 3. 关键代码

1. `OpenAI` 工具定义：

```python
TOOLS = [{
    "type": "function",
    "name": "bash",
    "description": "Run a shell command.",
    "parameters": {
        "type": "object",
        "properties": {
            "command": {"type": "string"},
        },
        "required": ["command"],
        "additionalProperties": False,
    },
    "strict": True,
}]
```

2. 找出模型请求的工具调用：

```python
tool_calls = [
    item for item in response.output
    if getattr(item, "type", None) == "function_call"
]
```

3. 回填工具输出：

```python
messages.append({
    "type": "function_call_output",
    "call_id": call.call_id,
    "output": output,
})
```

4. 解析模型传来的 `JSON` 参数：

```python
def parse_arguments(raw: str) -> dict:
    try:
        return json.loads(raw or "{}")
    except json.JSONDecodeError as e:
        return {"_error": f"Invalid JSON arguments: {e}"}
```

## 4. 我理解的流程

```text
user message
  -> client.responses.create(...)
  -> response.output
  -> messages.extend(as_input_item(item) for item in response.output)
  -> 找到 type == "function_call" 的输出项
  -> parse_arguments(call.arguments)
  -> run_bash(command)
  -> messages.append(function_call_output + call_id + output)
  -> 回到下一轮 responses.create(...)
  -> 没有 function_call 时返回 response.output_text
```

## 5. 与 `Anthropic` 版的差异

| 对比项 | `Anthropic` 版 | `OpenAI` 版 |
| --- | --- | --- |
| 模型调用 | `client.messages.create(...)` | `client.responses.create(...)` |
| 系统提示 | `system=SYSTEM` | `instructions=SYSTEM` |
| 历史输入 | `messages=messages` | `input=messages` |
| 工具请求 | `tool_use` | `function_call` |
| 工具结果 | `tool_result` | `function_call_output` |
| 调用对齐字段 | `tool_use_id` | `call_id` |
| 终止判断 | `response.stop_reason != "tool_use"` | `not tool_calls` |

## 6. 需要注意的风险

1. `OPENAI_MODEL` 示例值是 `gpt-5.5`，如果本地账号不可用，运行会在模型请求阶段失败。
2. `run_bash(command)` 仍然使用 `shell=True`，安全策略仍只是教学级字符串拦截。
3. 简报中没有看到真实 `function_call` 冒烟验证记录，所以不能确认当前账号和模型下的完整链路已经跑通。
